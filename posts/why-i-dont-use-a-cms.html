Why This Blog Doesn't Use a CMS

April 18, 2026 · 4 min read

This entire site is plain HTML, CSS, and a small amount of JavaScript. No database, no admin dashboard, no login page, no plugin updates to worry about at two in the morning. To publish a post, I copy a file, write in it, and add a link to it. That's the whole publishing pipeline.

What a CMS actually costs

A content management system solves real problems — multiple authors, scheduled publishing, a non-technical editor who needs a friendly interface. None of those problems exist here. What a CMS would add, for a one-person blog like this, is a server to maintain, a login to secure, and a whole extra piece of software that could break independently of anything I actually wrote.

A static file can't be hacked through a plugin vulnerability. It can't go down because a database ran out of disk space. It either loads, or the entire internet is down.

A static file can't be hacked through a plugin vulnerability, and it can't go down because a database filled up.

What it costs the other way

The honest tradeoff is that adding a post means opening a text editor instead of clicking "New Post" in a friendly dashboard, and there's no rich-text button for bold or italics — it's an actual HTML tag, typed by hand. For one person publishing a few essays a month, that's a trivial cost. It would stop being trivial the moment several people needed to publish here, or the moment posts needed scheduling, comments, or a search feature more advanced than filtering the homepage.

The actual filter

The question that decided it wasn't "which tool is more powerful," it was "what does this specific site actually need." A single-author blog that just needs to display finished essays doesn't need a content management system. It needs content, and a place to put it.
